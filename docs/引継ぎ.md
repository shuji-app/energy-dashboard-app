# エネルギー管理ダッシュボード — 引継ぎ.md

このドキュメントは、Claude Codeとの対話で `energy-dashboard-app` を一から作った際の
実装履歴・設計判断・データ品質問題の調査経緯をまとめたものです。今後このアプリに手を入れる際は
（Claude Codeによる作業を含め）必ず目を通してください。「なぜこうなっているか」が分かるように
書いています。

---

## 1. このアプリは何か

設備（吸収式冷凍機＋ボイラー）の日次エネルギーデータをもとにした、単一ページのエネルギー管理
ダッシュボードです。元データは `エネルギー.xlsx`（日付ごとの外気温・湿度、冷水/温水熱負荷、
購入電力量、重油使用量の記録）。以下の3モードを持ちます。

- **ダッシュボード**：日/月/年の期間切替、複数グラフ（項目・グラフ種別を自由に設定可能）、
  前期比較、予測（月内の残り日数・年度残り月数を推定）、KPIカード（電力量・重油・原油換算・
  デマンド・概算コストなど）。
- **データ入力**：カレンダーからの日次入力フォーム（前日/1週間前の参考値表示、バリデーション、
  Enterキーで保存→翌日へ移動、前日値コピー、メモ入力）、入力済み一覧のCSV的テーブル表示、
  メモ一覧、未入力日一覧、Excel再取込（データ更新）。
- **分析**：原単位トレンド、異常値検知（メモへの記録可）、号機別稼働バランス、目標比較、
  カレンダーヒートマップ、CO2トレンド、デマンド分析（ヒストグラム）、複数年度重ね表示。
  各グラフにPNG画像保存ボタンあり。

その他：設定（換算係数・単価・しきい値など全パラメータ変更可）、バックアップ/リストア
（JSON書き出し/読み込み）、ダークモード、印刷用レポート（`window.print()` + 印刷専用CSS）、
PWA対応（ホーム画面インストール・オフライン起動）。

## 2. アーキテクチャの前提（重要）

- **ビルドツールなし。** `app.jsx`は素のReact（グローバルの`React`/`ReactDOM`を直接使用、
  `const { useState, useEffect, useMemo, useRef, useCallback } = React;`）で書かれており、
  ES moduleのimport/exportは一切使いません。
- React 18.3.1 / ReactDOM 18.3.1 / Chart.js 4.4.4 / xlsx(SheetJS) 0.18.5 は全て
  `index.html`内でバージョン固定のCDN URLから読み込みます。
  - Chart.jsだけは cdnjs では404になるため **jsdelivr** (`cdn.jsdelivr.net/npm/chart.js@4.4.4/...`)
    を使っています。他のライブラリをcdnjsからjsdelivrに変える必要はありません。
- `build.js`は`app.jsx`をBabel（`@babel/preset-react`、classic runtime）でJSX→プレーンJSに
  変換して`app.compiled.js`に書き出すだけの薄いスクリプトです。
- データは初回`data/energy-data.js`（`window.ENERGY_DATA=[...]`）から読み込み、以後は
  `localStorage`に保存されます。バックエンドはありません。
- `ChartCanvas`という共通コンポーネントが全グラフの描画を担い、`useEffect`（依存配列は空）で
  Chart.jsインスタンスをマウントします。設定オブジェクトには関数（tooltipコールバック等）が
  含まれ単純比較できないため、diffの代わりに文字列化した`key`をpropとして渡し、内容が変わったら
  Reactに強制的にアンマウント→再マウントさせる方式を採っています（ダークモード切替時に
  `Chart.defaults`のグローバル設定を反映させるためにも、`darkMode`を必ず`key`に含めています）。

## 3. データについて

### 3.1 スキーマ
1日1レコード：日付、外気温、湿度、冷水熱負荷(GJ)、温水熱負荷(GJ)、購入電力量(kWh)、
重油使用量(L、冷凍機用/ボイラー用など号機別)。ここから`withDerived()`で
デマンド(kW)・合計熱負荷(GJ)等を導出します。

### 3.2 データ破損の調査と対応（重要・恒久対応済み）
元の`エネルギー.xlsx`のSheet1で、Excel行247〜488付近に**日付が非連続（後退・重複）で
ハードコードされた**（数式ではなく固定値として入力された、`.Formula == .Value2`で確認）
242行分のセルが見つかりました。

- 調査手順：全行の日付を`toDays()`的なヘルパーで差分チェックし、後退ジャンプが4箇所あることを
  特定 → 該当する242行の連続区間を特定。
- 当初はユーザーの判断で「ひとまず除外して進める」ことにしましたが、その後さらに精査した結果、
  242行のうち**91行（2024-12-01〜2025-03-01）は数値も妥当でかつ内部的に日付が連続していた**
  ため、この91行だけをデータセットに復元しました。残り151行は本当に破損しており、恒久的に
  除外しています（`data/excluded-rows.json`に理由付きで保管、アプリ本体は読み込まない）。
- 最終的な正データ：`data/energy-data.json`、721件、2024-04-01〜2026-03-31、この範囲に
  日付の欠落なし（ただし2025-10-01〜03、2025-12-01〜04、2026-01-01〜02など、合計9日分の
  小さな「本当に未入力」の欠落は残っています＝これはデータ破損ではなく単純未記入）。
- **再発防止策**：「データ更新」（Excel再取込）機能内に、取込中の行の日付が
  それまでに処理済みの最大日付より後退していたらその行をスキップするガード
  （`DataUpdateModal.handleFile`内）を追加済み。同じ破損が将来の再取込で紛れ込むのを防ぎます。

このデータの経緯があるため、**数値がおかしく見えても勝手に補正しないこと**。まず元データ
(`エネルギー.xlsx`本体、または`data/energy-data.json`)を確認し、必要ならユーザーに確認する
運用にしています（下記「6. 未解決の問題」も参照）。

### 3.3 localStorageキー一覧
| キー | 内容 |
|---|---|
| `LS_DATA_KEY` | 日次データ本体（配列。Excel再取込でマージ/upsert-by-date） |
| `LS_SETTINGS_KEY` | 設定（換算係数・単価・しきい値など） |
| `LS_VIEWSTATE_KEY` | 表示状態（mode/charts構成/選択中の期間など） |
| `LS_NOTES_KEY` | メモ（日付→テキストのマップ） |
| `LS_DARKMODE_KEY` | ダークモードON/OFF |

実際のキー文字列名は`app.jsx`冒頭の定数定義を参照してください。

### 3.4 設定項目（`DEFAULT_SETTINGS`）
契約電力(kW)、重油発熱量(MJ/L)、電力→MJ換算係数、電力CO2排出係数、重油CO2排出係数、
目標電力量(kWh)、目標重油使用量(L)、原油換算係数(GJ/kL)、異常値検知のσしきい値、
デマンド警報しきい値(%)、電力単価(円/kWh)、重油単価(円/L)。すべて設定画面から変更可能。

20. **ローカルファイルへの自動保存機能を追加**（File System Access API、設定画面の新セクション）。
    localStorageはブラウザのキャッシュ削除等で消えるリスクがある、という話の流れでユーザーが
    選んだ方式（他の選択肢：Firebase等クラウドDB連携、手動バックアップの定期運用）。
    - `window.showSaveFilePicker()`でユーザーがPC上のJSONファイルを1つ選択/新規作成すると、
      以後`data`/`settings`/`notes`のいずれかが変わるたびに、既存の「バックアップ」機能と
      同じペイロード形式（`buildBackupPayload()`に共通化）でそのファイルへ自動上書き保存する
      （`useEffect`の依存配列に`fileHandle`を含めることで、連携直後の初回保存も兼ねる）。
    - `FileSystemFileHandle`は`localStorage`に保存できない（構造化複製はできるがJSON化できない）
      ため、IndexedDB（`idbOpen`/`idbSaveHandle`/`idbLoadHandle`/`idbClearHandle`、DB名
      `energyAppFS`）に保管し、次回起動時も同じファイルへ書き込み続けられるようにした。
    - 再起動時は`handle.queryPermission()`で権限を確認し、`granted`ならそのまま自動保存を再開。
      それ以外（`prompt`）の場合はブラウザの制約上ユーザー操作なしに`requestPermission()`を
      呼べないため、「再連携する」ボタンを表示してユーザーのクリックをトリガーに許可を取り直す。
    - Chromium系ブラウザのみ対応（`"showSaveFilePicker" in window`で機能検出、非対応ブラウザには
      その旨を表示するのみで他の設定項目には影響しない）。
21. **Firebase Realtime Databaseによる複数端末（iPad＋PC等）リアルタイム同期を追加**。
    「iPadとデスクトップで同期を取りながら使いたい」という要望に対し、`ac-check-app`が既に
    同じ用途（iPad+PCのリアルタイム共有）でFirebase Realtime Databaseを使っており実績があった
    ため、同じ設計パターンを移植した（`ac-check-app`の`app.jsx`を事前に調査し、実装を忠実に
    踏襲）。ただし書き込み方式は本アプリ向けに整理し直している（詳細は下記）。
    - **SDK**：`firebase-app-compat.js` + `firebase-database-compat.js`（v10.14.1、CDN、
      `index.html`で`app.compiled.js`より前に読み込み）。modular SDKではなくcompat SDK
      （`window.firebase.initializeApp(...)`スタイル）を使用（ac-check-appと同じ）。
    - **`firebaseConfig`**：`app.jsx`冒頭（`DEFAULT_CHARTS`の直後）にプレースホルダー値
      （`apiKey: "YOUR_API_KEY"`等）で定義。実際の値は、ユーザー自身がFirebaseコンソールで
      新規プロジェクトを作成して取得する必要があるため（Googleアカウントでのログイン・
      プロジェクト作成はエージェントが代行できない操作）、README.mdに手順を追記し、
      ユーザー側で`firebaseConfig`を書き換えて`npm run build`してもらう運用とした。
      `apiKey !== "YOUR_API_KEY"`かつ`window.firebase`が存在する場合のみ`db`が有効になり、
      それ以外（未設定・CDN読み込み失敗等）は`db = null`となって同期処理は全てスキップされ、
      アプリは今まで通りlocalStorageのみで動作する（ac-check-appと同一のガード条件）。
    - **Realtime DBのパス構成**：`energyData/{date}`（1日1レコード）、`settings`（設定オブジェクト
      まるごと）、`notes/{date}`（メモ）の3ノード。ac-check-appの「点検回」のような入れ子構造は
      不要なため、よりシンプルなフラット構成にした。
    - **読み取り（他端末の変更を受信）**：`data`/`settings`/`notes`それぞれに対して、マウント時
      （空の依存配列）に`ref.on("value", handler)`を張る3本の`useEffect`。スナップショットが
      存在すれば`setData`/`setSettings`/`setNotes`とlocalStorageキャッシュを更新、存在しなければ
      （＝新規Firebaseプロジェクトでまだ何も無い場合）その時点のローカルデータを初期シードとして
      アップロードする（`fbBulkSaveData(data)`等）。
    - **書き込み（自分の変更を送信）**：ac-check-appの設定まわりは「stateの変化を監視する
      useEffectで書き込む」方式だったが、これは「リモートから受信→ローカルstate更新→
      それを監視するeffectが再度リモートへ書き込む」という不要な往復（エコー書き込み）が
      発生し得る設計だったため、本アプリでは踏襲しなかった。代わりに、実際の保存操作を行う
      各ハンドラ（`handleSaveRecord`/`handleDeleteRecord`/`handleSaveNote`/`handleSaveSettings`/
      `handleImportData`/`handleImportBackup`）の中から直接
      `fbSaveRecord`/`fbDeleteRecord`/`fbSaveNote`/`fbSaveSettings`/`fbBulkSaveData`/`fbReplaceAll`
      を呼ぶ構成にし、"読み取り用effect"と"書き込み"を完全に分離した（読み取り側のsetStateが
      書き込みを誘発することは構造的に起こらない）。
      - `fbSaveRecord`/`fbDeleteRecord`/`fbSaveNote`：1日単位の`set()`/`remove()`（通常の保存・削除操作）。
      - `fbBulkSaveData`：Excel再取込（`handleImportData`）で追加/更新された行だけを
        `db.ref().update({...})`でまとめて送る（変更の無い行は送らない）。
      - `fbReplaceAll`：「バックアップから復元」時専用。energyData/settings/notesの3ノードを
        丸ごと置き換える（復元＝「これが正」の全置換という意味合いのため、増分updateではなく
        `set`相当の完全上書きにしている）。
    - ヘッダーの「データ最終日」の右に、`db`が有効なときだけ「🔥 同期中」バッジを表示
      （`db`はモジュールスコープの定数のため、追加のstateなしで判定可能）。
    - デバウンス処理は入れていない（ac-check-appと同じ判断：レコード保存はボタン押下等の
      明示的操作の時だけ発生し、キー入力のたびに書き込まれるものではないため）。
    - エラーハンドリングは`db.ref(...).set()`等をfire-and-forgetにしている点もac-check-appを踏襲
      （書き込み失敗時のUI通知は無し。将来的に必要になれば追加を検討）。

## 5. 発見・修正した重要な不具合

### 5.1 Service Workerのキャッシュ不具合（★最重要・再発しやすいので必読）
`sw.js`は当初、**全リクエストをキャッシュ優先**（`return cached || fetchPromise`、
ネットワーク応答は次回訪問時のキャッシュ更新にしか使わない）で実装していました。これだと
同一オリジンのアプリ本体ファイル（`app.compiled.js`等）は、**コードを修正して`npm run build`
しても、ブラウザには常に「1つ前のバージョン」が表示され続ける**という重大な不具合がありました
（新しいレスポンスはキャッシュに保存されるだけで、今回の表示には使われないため）。

**修正**：`sw.js`のfetchハンドラを分割し、
- 同一オリジン（アプリ本体）→ **ネットワーク優先**、失敗時のみキャッシュにフォールバック
- クロスオリジン（バージョン固定CDN）→ 従来通りキャッシュ優先（不変URLなので問題なし）

に変更。あわせて`CACHE_NAME`を`"energy-app-cache-v1"`→`"energy-app-cache-v2"`に上げ、
`activate`イベントのクリーンアップ処理で旧キャッシュを確実に破棄するようにしました。

この不具合は、実際に「概算コスト」KPIカードが（コード上は正しく実装されているのに）
ブラウザに表示されないという形で発現し、原因特定に時間がかかりました（`bash grep`や
`fetch(...,{cache:'no-store'})` + `.includes()`によるテキスト検索が、日本語の多バイト文字や
Service Workerのインターセプトの影響で信頼できない結果を返したため——**このアプリのデバッグでは、
ファイル内容の文字列検索より、実際にブラウザでレンダリングされたDOMを
`document.querySelectorAll(...).textContent`等で直接確認する方が確実**です）。

**今後コードを修正しても画面に反映されない場合**は、まずこのService Workerキャッシュを疑い、
ブラウザの開発者ツールでService Worker登録解除＋キャッシュ全削除を試してください。

### 5.2 グラフ・予測関連
- **前期比較ON時に予測（点線）が表示されない**：`buildMainChartConfig`の予測オーバーレイ処理が
  非比較モードの分岐にしかなかったため。共通の`pushForecastDataset`関数に切り出し、比較・
  非比較どちらの分岐からも呼ぶよう修正。
- **月/年の着地予測が「実績合計より少ない」など不可能な値になる**：`computeFiscalYearForecast`の
  旧ロジックは「日数カウントで完了/未完了を判定」する簡易ヒューリスティックで、複数の月に
  未入力日があると最初の1ヶ月しか未完了として扱わず、他の未完了月をまるごと合計から
  落としてしまうバグがありました。`latestDate`が属する月のインデックス(`currentIdx`)を基準に、
  「完全に過去の月は欠落込みでそのまま合計」「現在進行中の1ヶ月だけ日単位で予測」
  「未来の月は推定」と明確に3分類するロジックに書き換えて修正。
- **年表示でも前期比較のスタイルが残る**：`buildMainChartConfig`呼び出し側で
  `compareEnabled && view !== "year"`を強制することで解消。

### 5.3 UI/レイアウト
- `.btn-ghost`（白文字）が明るい背景のモーダル内で使われて見えなくなる不具合（設定モーダルの
  キャンセルボタン、バックアップボタン、全画面モーダルの閉じるボタン等）→ 明るいモーダル内は
  すべて通常の`.btn`に統一。
- KPIグラフパネルの高さが固定されていたため、系列数増加や予測バナー表示時にグラフが
  はみ出す/切れる → 高さ固定を `.chart-canvas-wrap`（キャンバス部分のみ）に限定し、
  パネル全体は可変にした。

### 5.4 データ入力・再取込
- 「データ更新」で再取込しても画面が変わらないように見える不具合 → マージ/永続化処理自体は
  正しく動作していたが、**表示中の期間（選択中の年月/年度）が新しい最新日付へ自動的に
  切り替わらなかった**ことが原因。`handleImportData`内で、取込後に選択期間を最新日付へ
  ジャンプさせる処理を追加。
- 異常値検知のσ計算が、非数値の温度/値セルが混入した際に`NaN`を返して検知件数が黙って0に
  なる不具合 → `AnalysisView`の異常値`useMemo`と`App`の`recentAlerts`の両方に
  `!isNaN(...)`の明示チェックを追加。
